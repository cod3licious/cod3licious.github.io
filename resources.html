When it comes to programming in general, a process that works well for me is:

  1. Understand the problem & develop a plan: Turn off your computer, take out pen and paper and sketch the solution: What steps does it take to solve the problem? How can they be subdivided into reusable functional units? How do the individual components work together? What are their inputs and outputs? What data structures could simplify things?
  2. Write documentation and comments: Once you've developed a plan for how to solve your problem, turn your computer back on and open your code editor of choice. Start with the main function and write comments for all the steps that need to be executed to solve your problem. If one step should be moved to a separate utility function, write the skeleton of the function and its docstring describing what the function does and what its inputs and return values are. For these utility functions you can either add comments describing the individual steps now or when you start coding the main function before you dive into the subroutines.
  3. Code: After you've sketched out all the steps, start with the actual coding, i.e., fill your comment skeleton with life. While coding, in addition to the comments outlining the high-level steps for solving the problem, it often also makes sense to document any unusual design choices. For example, if you couldn't implement something in a straightforward way, because your data is messy in some places and this would cause an error, explain this in a comment, otherwise someone will later wonder why this was done this way instead of the more efficient solution and might end up wasting time rediscovering your initial problems. Furthermore, comments are very helpful to remind yourself about things that need to be changed in the future, e.g., "# TODO: once the data collection pipeline has been fixed, this code can be simplified to ...".
  4. Refactoring: Check your whole program again: Could your code be improved somewhere? For example, do you have code blocks that are copy & pasted and could be put into a separate function to avoid later having to change the same code in multiple places? Does any one function seem particularly long and would it make sense to move pieces of it to a separate function?

By first being clear about what you want to accomplish and sketching out the steps in the comments, your code will have a much cleaner structure and you ensure that it contains all the necessary comments for someone else to understand what you were trying to do here (where this someone else may often be you two weeks later - trust me, you'll thank your past self later ;-)).
